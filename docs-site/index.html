Overview

What Wed Plan helps you do

Track the budget that actually matters

Wed Plan focuses on wedding spending, vendor decisions, guest planning, and the timeline for the day itself. It is designed for couples using their phones, so screens stay narrow, actions stay close by, and the most important numbers remain visible.

Core concepts

  • Groups organise related expenses like Food, Venue, or Attire.
  • Expenses are individual planning items such as Venue or Photographer.
  • Vendor options let you compare quotes, notes, contacts, and payment status.
  • Guests track RSVPs, parties, households, and special requirements.
  • Timeline keeps the wedding day plan in one place.

Getting started

Run the app locally in a few minutes

Docker setup

  1. Clone the repository.
  2. Run docker compose up -d --build.
  3. Open http://localhost:3000.
  4. The database is created automatically on first start.

Local development

  1. Install dependencies with npm ci.
  2. Start the app with npm run dev.
  3. Open http://localhost:3000.
  4. Use the bottom navigation to move between dashboard, timeline, guests, and settings.

Recommended first-run workflow

  1. Open Settings and add the couple names, wedding date, and total budget.
  2. Create or rename groups to match the areas you are budgeting for.
  3. Add expenses inside each group.
  4. Add vendor options for each expense and mark the preferred choice as Selected.
  5. Add guests and households once the budget is taking shape.
  6. Build the wedding day timeline as plans become firm.

Feature guides

Use every part of the app with confidence

Dashboard and expense planning

  • See the total budget, committed amount, paid amount, and outstanding spend.
  • Create expense groups and add expenses directly from the dashboard.
  • Collapse groups to keep the mobile view manageable.
  • Reorder expenses within a group with drag handles.
  • Watch reminders for deposit and final payment dates on selected vendors.

Expense details and vendor comparison

  • Open any expense to set a budget allocation and move it to another group.
  • Add multiple vendor options with notes, pros, cons, contact details, and payment dates.
  • Mark the preferred option as Selected and later toggle it to Booked.
  • Use the compare view when you have two or more options for the same expense.
  • Track deposit paid, total paid, and how much remains against the expense budget.

Guest list and households

  • Track guests by party, attendance type, and RSVP status.
  • Create households for shared addresses and grouped invitations.
  • Record plus-ones, dietary requirements, allergies, accessibility needs, and table assignments.
  • Filter and search the guest list as it grows.
  • Export guest data when you need to share it elsewhere.

Timeline and settings

  • Split the day into phases and add timeline moments for each one.
  • Store a wedding date so the timeline is anchored to the right day.
  • Switch between light, dark, and system theme modes.
  • Keep the app personalised with couple names and a total budget in settings.

Configuration

Runtime and data configuration

Environment variables

The app currently supports one runtime setting:

  • DATABASE_PATH — overrides where the SQLite database file is stored.

In Docker, the default path is /data/wedding.db. In local development, the default path is ./data/wedding.db.

Data and backups

  • The app stores data in a SQLite database.
  • Docker Compose mounts the database inside the named volume wed-plan-data.
  • Back up the database file before upgrades if the data matters.
  • Stop the app before restoring a backup, copy the database files back into place, then confirm the files are writable by the app process before starting it again.

Docs site layout

The documentation site is intentionally separate from the app. Static files live in docs-site/, so the docs can deploy to GitHub Pages without changing the main Next.js runtime.

Deployment

Run the app yourself and publish the docs automatically

Deploy the app

  1. Provision a server that can run Docker.
  2. Clone the repository and keep the database on persistent storage.
  3. Run docker compose up -d --build.
  4. Point your reverse proxy or DNS at the server if you need public access.
  5. Update by pulling new code and rebuilding the Docker image.

Deploy the docs site

  1. The GitHub Actions workflow packages the docs-site/ directory.
  2. On pushes to main or manual dispatch, it publishes that static content to GitHub Pages.
  3. In repository settings, set the Pages source to GitHub Actions.

Preview the docs locally

cd docs-site
python3 -m http.server 8000

Then open http://localhost:8000.

Troubleshooting

Common problems and quick fixes

The app is not reachable

  • Check docker compose ps if you are using Docker.
  • Review container logs with docker logs wed-plan --tail 200.
  • Confirm the app is listening on port 3000.

The latest code is not showing up

  • Rebuild the container with docker compose up -d --build.
  • If needed, stop and restart the stack with docker compose down followed by docker compose up -d --build.

The database path is wrong

  • Set DATABASE_PATH to a writable location.
  • Make sure the target directory exists or can be created by the app process.
  • Preserve the database files when replacing containers.

GitHub Pages is not updating

  • Confirm Pages is configured to use GitHub Actions.
  • Check the Deploy Docs to GitHub Pages workflow run for failures.
  • Make sure your docs changes are committed on the branch that triggers the workflow.

FAQ

Short answers to common questions

Can I use Wed Plan without Docker?

Yes. Install dependencies with npm ci and run the app with npm run dev.

Does the app support multiple vendor quotes per expense?

Yes. Each expense can hold multiple vendor options, and one option can be marked as selected or booked.

Where is my data stored?

In a SQLite database file. The location is controlled by DATABASE_PATH or the default data directory.

Is the docs site tied to the Next.js app build?

No. The docs site is plain static HTML and CSS under docs-site/, and it deploys separately to GitHub Pages.

What should I back up before upgrades?

Back up the SQLite database file and its related WAL/SHM files if they exist.
